# AWS DCO 인턴십 직무 이해를 위한 생성형 AI 활용 포트폴리오

## 프로젝트 소개

이 프로젝트는 공개된 DCO 직무 정보와 교육용 샘플 데이터를 바탕으로
DCO 직무의 기본 용어, 교육용 로그 분석, 장애 상황 문서화와 인수인계 흐름을 학습한 결과물입니다.

생성형 AI를 단순히 답을 생성하는 도구로 사용하지 않고,
용어 정리, 문서 구조화, 로그 분석, Python 코드 작성, 보고서 초안 작성과 결과 검증을 돕는 도구로 활용했습니다.

이 프로젝트는 실제 AWS 내부 시스템, 실제 운영 로그, 실제 Ticket 또는 실제 데이터센터 장비를 대상으로 수행한 프로젝트가 아닙니다.

## 생성형 AI 활용 원칙

* 생성형 AI의 답변은 최종 정답이 아니라 검토가 필요한 초안으로 사용했습니다.
* AI가 작성한 장비명, 시간, Ticket ID와 사건 내용을 원본 로그와 비교했습니다.
* 로그에서 직접 확인되는 사실과 가능한 추정을 구분했습니다.
* 로그만으로 확인할 수 없는 Root Cause는 확정하지 않았습니다.
* Ticket 우선순위는 AI가 대신 결정하지 않고, 학생이 근거를 비교하여 직접 판단했습니다.
* 실제 계정정보, 실제 장비정보, 실제 Ticket과 실제 AWS 내부정보를 AI에 입력하지 않았습니다.
* 모든 로그, 장비명, Ticket과 IP 주소는 교육용 샘플 데이터로 구성했습니다.

## 배운 점

### 1. AI 협업 및 검증 프로세스
* **작성 시간 단축 및 업무 효율화**: 생성형 AI를 활용하면 로그 요약, 코드 작성과 문서 초안 작성 시간을 획기적으로 줄일 수 있습니다.
* **철저한 비교 검증 필수**: AI가 빠르게 결과를 생성하더라도, 장비명, 시간대, 특정 에러 건수 등 왜곡된 내용(Hallucination)이 없는지 원본 자료와 비교하는 검증 과정이 반드시 수반되어야 합니다.
* **학습 및 메타-인지 아카이빙**: 기술 결과물뿐 아니라 AI를 어떤 목적으로 사용하고, 사람이 직접 무엇을 검증하고 수정했는지를 기록하고 통제하는 능력이 중요합니다.

### 2. 인프라 실무 데이터 분석 및 문서화 기법
* **사실과 추정의 엄격한 분리**: 로그 분석에서는 직접 확인된 사실(Fact), 이를 토대로 한 가능한 추정(Hypothesis), 그리고 로그상 부재한 미확인 정보(Unknown)를 명확히 구분하여 작성해야 합니다.
* **복구와 RCA(근본 원인 분석)의 독립성**: 장애가 복구되었다는 사실만으로 정확한 Root Cause가 확인된 것은 아닙니다. 패치코드 교체와 커넥터 청소를 병행 조치했을 경우, 단일 원인을 단정 짓지 않고 모든 가설을 검토해야 합니다.
* **문서 작성의 목적성**: Incident Report는 사건의 흐름과 조치 결과를 비전공자나 타 팀원도 쉽게 이해할 수 있도록 구조화하여 작성해야 하며, Handover Memo는 교대 전에 진행 중인 상황과 다음 확인 사항(Action Item)을 후임자가 즉시 실행할 수 있도록 실무적으로 전달해야 합니다.

### 3. 리스크 기반 우선순위(Triage) 판단
* **Severity(심각도)와 Priority(우선순위)의 분리**: Severity는 사건 자체의 기계적 심각도(예: ERROR, WARNING)이고, Priority는 현업에서 비즈니스 영향력을 분석하여 무엇을 먼저 조치할 것인지 결정하는 현실적인 처리 순서입니다.
* **종합적 리스크 판단**: Ticket 우선순위는 단순 로그 레벨에만 의존해서는 안 되며, 서비스에 미치는 영향, 반복지속 여부, 이중화 소실 위험성(예: PDU 입력 피드 한쪽 불량 등), 영향 범위(서버 대수)와 복구 여부를 종합적으로 비교해 판단해야 합니다.

### 4. 실무 도구 및 호환성 트러블슈팅
* **Excel 호환성을 위한 인코딩 관리**: UTF-8 형식의 CSV 파일은 Windows Excel에서 열 때 한글 깨짐 현상이 빈번히 발생합니다. 이를 위해 파일의 가장 처음에 BOM 표식을 붙이는 `utf-8-sig` 형식으로 저장해야 인코딩 깨짐을 원천 차단할 수 있음을 체득했습니다.
* **동시성 잠금(File Lock) 예외 처리**: 사용자가 Excel 등의 도구로 파일을 점유하고 있을 때 스크립트를 재실행하면 `PermissionError`가 발생합니다. 이에 대응하여 우회 Fallback 파일 생성 및 잠금에 대한 사용자 안내 기능을 구성하는 안정적인 소프트웨어 개발 예외 처리가 필요함을 실습을 통해 배웠습니다.

## 프로젝트 범위와 제한

이 저장소는 DCO 인턴십 직무를 이해하기 위한 학습형 포트폴리오입니다.

* 실제 AWS DCO 업무 수행 능력을 증명하는 자료가 아닙니다.
* 실제 데이터센터 운영 절차나 내부 SOP를 포함하지 않습니다.
* 실제 장애 대응, 장비 교체 또는 운영 자동화를 수행하지 않습니다.
* 공개 직무 맥락과 교육용 샘플 데이터를 이용한 학습 결과만 포함합니다.
