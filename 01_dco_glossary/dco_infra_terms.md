## 2. 내가 만든 DCO 인프라 용어 정의

**Rack** : 서버들을 세로로 층층이 꽂아 두는 철제 캐비닛으로, 데이터센터에서 장비 위치를 말할 때 기준이 되는 단위다.

**Server :** 랙에 꽂혀 24시간 쉬지 않고 돌아가는 업무용 컴퓨터로, 점검·부품 교체 등 대부분의 작업 대상이 된다.

**NIC :** 서버가 네트워크에 연결되도록 해 주는 통신 부품으로, 이게 문제면 서버가 멀쩡해도 외부와 통신이 안 될 수 있다.

**ToR Switch** : 랙 위쪽에 설치되어 그 랙의 서버들을 묶어 바깥 네트워크로 연결하는 장비로, 문제가 생기면 랙 전체가 영향을 받을 수 있다.

**Link Down** : 네트워크 연결 자체가 끊어졌다는 신호다. 원인은 케이블, 부품, 반대편 장비 등 여러 가지일 수 있어 단정할 수 없다.

**CRC error** : 데이터가 도착은 했지만 검사해 보니 내용이 깨져 있다는 신호로, 연결은 살아 있지만 품질에 문제가 있다는 뜻이다. 전송 경로의 물리적 문제인 경우가 많다고 알려져 있지만 설정 문제 등 다른 원인도 있을 수 있다.

**SSD failure** : 서버의 저장 부품이 고장 난 상태로, 부품 교체 작업으로 이어지는 흔한 고장 유형이다.

**Fan Alert** : 서버 내부 냉각 팬에 이상이 있다는 경고로, 방치하면 온도 상승으로 번질 수 있는 예고 신호다.

## 3. 헷갈리는 용어 비교

#### [비교 1]

**내가 비교한 용어:** Link Down vs CRC error

**내가 이해한 차이:** Link Down은 연결 통로 자체가 끊어진 것이고, CRC error는 통로는 살아 있지만 도착한 데이터가 깨져 있는 것이다. "Down은 끊김, CRC는 깨짐"으로 구분한다.

#### [비교 2]

**내가 비교한 용어:** Packet Loss(패킷 유실) vs CRC error

**내가 이해한 차이:** 패킷 유실은 데이터가 아예 도착하지 않은 것이고, CRC error는 도착은 했지만 검사해 보니 내용이 깨져 있는 것이다. 처음에는 둘 다 "네트워크 문제"로 묶어서 생각했는데, 서로 다른 현상이었다. 유실은 네트워크 혼잡 등으로 장비가 패킷을 버리는 경우처럼 물리 고장이 아닌 원인도 흔한 반면, CRC 오류는 "전송 중에 데이터가 실제로 변형됐다"는 증거라서 신호가 지나가는 물리 구간(케이블, 커넥터 등)을 먼저 확인하는 단서로 쓰이는 경우가 많다고 알려져 있다. 다만 설정 불일치 같은 다른 원인도 있을 수 있어 단정할 수는 없다.

**기억법:** 안 온 것(유실) / 안 이어진 것(Link Down) / 깨져서 온 것(CRC) — 세 가지는 서로 다른 현상이다.

## 5. AI 답변을 보고 내가 수정한 점

- **실제 정보처럼 보일 수 있어 삭제하거나 수정한 표현:** 초안을 옮기면서 실제 장비명, 티켓 번호, IP 주소가 생성되지 않았는지 확인했고, 단정형으로 읽힐 수 있는 문장에 "~라고 알려져 있다", "~일 수 있다" 같은 표현이 유지됐는지 점검했다. 점검 결과 삭제할 내용은 없었다.
- **너무 단정적이라 바꾼 표현:** AI의 처음 설명에서 CRC 오류가 물리적 문제처럼 읽혔는데, 패킷 유실 같은 비물리적 원인도 있지 않은지 의문이 들어 다시 질문했다. 그 결과 "CRC 오류는 물리적 문제다"가 아니라 "전송 중 데이터가 손상됐다는 증거이며, 물리 구간 문제인 경우가 많지만 설정 문제 등 다른 원인도 있어 단정할 수 없다"로 이해를 수정했다.
- **아직 헷갈리는 용어 / 다시 공부하고 싶은 용어:** CRC error. 개념 구분은 됐지만, 설정 불일치처럼 물리 구간이 아닌 원인으로 CRC 오류가 생기는 구체적인 상황은 더 공부하고 싶다.

## 6. 오늘의 느낀 점

- **오늘 처음 알게 된 용어는:** Link Down과 CRC error. 특히 두 용어와 패킷 유실이 서로 다른 현상(끊김 / 깨짐 / 안 옴)이라는 것을 구분하게 됐다.
- **가장 헷갈렸던 용어는:** CRC error. 패킷 유실과 같은 "네트워크 문제"로 묶어서 생각하고 있었는데, 도착하지 않은 것(유실)과 깨져서 도착한 것(CRC)이 다른 현상이라는 것을 질문을 통해 구분하게 됐다.
- **AI 답변을 사용할 때 조심해야 할 점은:** AI 설명이 그럴듯해 보여도 의문이 들면 그냥 넘기지 말고 되물어야 한다는 것. 실제로 CRC 오류 설명에 의문을 갖고 질문했더니 처음 설명보다 더 정확한 이해에 도달했다. AI의 첫 답변을 정답으로 받아들이지 않는 것이 중요하다.
- **다음 차시에서 더 알고 싶은 내용은:** 에러 상황이 발생했을 때 실제로 어떤 순서로 대응하는지. 오늘은 장애 신호의 의미를 배웠으니, 다음에는 신호를 받은 뒤의 대응 흐름(확인 → 조치 → Escalation 판단 → 기록)을 알고 싶다.
